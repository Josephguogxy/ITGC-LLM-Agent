System Overview

What the ITGC agent system does in operation

The framework is organized around one practical goal: keep a network of DER-rich PDNs operating safely and economically under persistent uncertainty. Instead of treating each optimization step as an isolated solve, the system maintains a structured operating memory, translates strategic intent into executable bridge variables, and uses verification plus HCII review to keep decisions aligned with network constraints and service priorities.

Long-short timescale coordination

The ITGC planner reasons at a strategic horizon, while the orchestrator and local PDN stack act at the rolling dispatch horizon. Bridge variables carry long-horizon goals into short-horizon execution without breaking feasibility.

Closed-loop safety control

Verification is not an afterthought. Candidate decisions are checked against operating limits, acceptance rules, and recent failure signals before execution is approved.

Memory-driven evolution

The system persists successful cycles, failed cycles, context patterns, and workflow checkpoints to file. Future cycles reuse that knowledge to adapt objectives, warm-start dispatch, tighten safety, and avoid repeating known bad decisions.

Module Map

Functional modules and how each one contributes

The codebase is arranged as a readable operating stack. Each module owns a clear role, exposes structured outputs, and interacts with memory in a specific way.

ITGC Planner

The ITGC module maintains the strategic state of the system. It combines current operating intent, historical feedback, and memory-derived policy bias to produce a long-horizon plan that can guide subsequent rolling coordination.

  • Primary files: src/agent/itgc_agent.py, src/models/long_term.py, src/optimization/long_term/
  • Main outputs: updated strategic weights, long-horizon plan z_k, planning diagnostics
  • Memory usage: reads policy memory, accepted cases, failure pressure, and preferred posture
  • Why it matters: keeps weekly or day-ahead intent coherent even when local conditions change quickly
Planning Logic

ITGC does more than request a solve. It adapts weights, budgets, and risk posture based on what the system learned in previous cycles, then emits a plan that shapes the feasible behavior of downstream PDN agents.

Orchestrator Scheduler

The orchestrator converts strategic intent into a runtime workflow. It sequences sub-agents, constructs cycle plans, allocates checkpoints, and records rollback triggers that keep the closed loop manageable under stress.

  • Primary files: src/agent/orchestrator_agent.py, src/agent/framework.py
  • Main outputs: cycle plan, checkpoint map, task ordering, rollback hints
  • Memory usage: reads checkpoint history, warm-start availability, and failure tags
  • Why it matters: it turns the system from a collection of roles into a reliable runtime loop
Workflow Logic

By retaining workflow checkpoints and execution traces, the orchestrator can restart from feasible intermediate states instead of rebuilding the cycle from scratch.

Local PDN Agent Stack

Each PDN is handled through a local stack that updates state estimates, prepares localized forecasts, and translates bridge envelopes into short-term operational variables. This is the layer where local system conditions become executable inputs to optimization and verification.

  • Primary files: src/agent/dse_agent.py, src/models/short_term.py, src/models/types.py
  • Main outputs: local state/context, forecasts, dispatch-ready problem inputs
  • Memory usage: writes current context, alarms, and trigger patterns into state memory
  • Why it matters: it connects network reality to every later decision in the loop
Local Adaptation

The stack is where heterogeneous PDNs remain distinguishable. Different PV, EV, storage, and demand conditions are preserved instead of being flattened into one aggregate.

Optimization and Dispatch

The optimization layer transforms bridge variables and local state into a solvable dispatch problem. It supports backend abstraction, warm starts from checkpoint memory, and distributed coordination through ADMM-style consensus updates.

  • Primary files: src/optimization/builder.py, src/agent/optimization_agent.py, src/optimization/short_term/admm_dispatch.py
  • Main outputs: short-term decision X_k, convergence traces, backend status
  • Memory usage: reads checkpoint-derived warm starts and reusable profiles
  • Why it matters: makes closed-loop coordination computationally practical at multi-PDN scale
Numerical Core

Warm starts and prior feasible patterns help the dispatch layer converge faster and avoid repeating solve trajectories that previously led to failure or rejection.

Verification Layer

Verification checks whether a proposed first-step action is safe enough to execute. It evaluates network stress, power balance, reserve posture, and acceptance thresholds, then returns a structured decision signal that drives approval or rollback.

  • Primary files: src/agent/verification_agent.py
  • Main outputs: acceptance flag, reason codes, violation summary, review package
  • Memory usage: reads failure memory and policy bias to tighten safety after repeated bad cases
  • Why it matters: it prevents the system from turning a locally attractive plan into an unsafe execution
Safety Gate

Verification becomes stricter when similar contexts have failed recently, which makes the system genuinely adaptive instead of mechanically repeating the same acceptance rule.

HCII Agent

The Human-Computer Interaction Interface agent converts structured cycle outputs into operator-facing rationale, highlights risk, and supports a transparent approval workflow. It is also the point where the LLM can translate machine state into actionable guidance.

  • Primary files: src/agent/hcii_agent.py, src/agent/prompts.py, src/llm/
  • Main outputs: operator summary, approval recommendation, explanation package
  • Memory usage: reads recent accepted and failed cycles to contextualize advice
  • Why it matters: it makes the agent system interpretable, reviewable, and deployable in supervisory settings
LLM Interface

The HCII layer does not replace optimization. It explains structured outcomes, preserves role boundaries, and helps the human-facing side of the loop remain consistent.

Memory Layer

The memory system turns the framework into an iterative evolution engine. It stores state/context patterns, policy updates, accepted cases, failure records, and workflow checkpoints on disk and retrieves them at the beginning of each cycle.

  • Primary files: src/agent/memory.py, memory_store/agent_memory.json
  • Main outputs: retrieval package, write-back records, updated policy memory
  • Memory usage: this module is the persistence layer itself
  • Why it matters: it lets the system improve behavior over time without retraining model weights
Persistent Knowledge

Success memory improves reuse, failure memory improves avoidance, and checkpoint memory improves solver efficiency. Together they make future cycles better prepared from the start.

Runtime Chain

The full closed-loop operating logic

A single cycle is not just “plan, solve, act.” It is a structured long-short timescale loop in which memory retrieval, bridge generation, dispatch, safety review, execution, and write-back all influence the next cycle.

State / Context

Historical states, alarms, exogenous context, and trigger patterns.

Policy / Goal

Weights, risk budgets, acceptance thresholds, and preferred posture.

Success Cases

Accepted cycles with feasible plans, bridge variables, and feedback signatures.

Failure / Rollback

Rejected cycles, failure labels, rollback reasons, and future avoidance cues.

Checkpoint / Workflow

Warm starts, feasible checkpoints, solver traces, and runtime templates.

Why bridge variables matter

The bridge package is the contract between strategic planning and rolling control. It keeps local agents aligned with long-horizon resource commitments without forcing them to solve the full strategic problem at every short-term step.

Why verification stays separate

Optimization can produce a mathematically appealing answer that still violates practical safety posture. The verification stage isolates that judgment and lets failure memory dynamically tighten future acceptance conditions.

Why write-back changes future behavior

After every accepted or rejected cycle, the memory system updates what the framework believes about risk, solver initialization, and successful operating patterns. This is the source of the system’s iterative improvement.

Multi-PDN Example

A complete example of one coordinated operating cycle

The example below illustrates how the closed loop behaves when a network of PDNs faces simultaneous renewable fluctuation and EV-driven demand pressure. The goal is to show how the modules cooperate in sequence, not just what files exist.

8 PDNs 15-min dispatch interval 60-min rolling horizon PV drop + EV surge + import stress

Representative network state

PDN-2

High EV departure demand, moderate storage state, rising local price exposure.

PDN-4

Sharp PV reduction, import dependency increasing, bridge reserve close to lower bound.

PDN-6

V2G-capable EV fleet available, fast response potential, good candidate for flexibility support.

System-level stress

Net load ramps upward, aggregate import envelope tightens, and verification must guard against overdraw.

Step 1. Retrieve relevant operating memory

The cycle begins by loading file-backed memory. The framework checks whether similar high-ramp contexts recently succeeded, whether failures occurred under tight import conditions, and whether a feasible checkpoint exists for a comparable dispatch profile. 

{
  "context_match": "renewable_drop + ev_surge",
  "failure_pressure": 0.18,
  "success_credit": 0.62,
  "warm_start_available": true
}

Step 2. Update strategic plan

ITGC adjusts the objective vector toward resilience and peak-shaving because the retrieved context indicates higher ramp stress. It then produces a long-horizon plan that slightly increases reserve posture while preserving EV service quality. 

{
  "weights": {"cost": 0.31, "resilience": 0.34, "service": 0.20, "carbon": 0.15},
  "plan_summary": {"reserve_bias": 0.07, "target_import_cap": 18.4}
}

Step 3. Build per-PDN bridge envelopes

The strategic plan is translated into bridge variables so that each PDN receives an executable envelope for import, storage usage, V2G support, and reserve contribution. This is how strategic intent becomes operational structure. 

{
  "PDN-2": {"import_cap": 3.6, "battery_floor": 0.28},
  "PDN-4": {"import_cap": 2.9, "shed_penalty": 9.5},
  "PDN-6": {"v2g_support": 1.4, "reserve_obligation": 0.5}
}

Step 4. Coordinate distributed dispatch

The orchestrator launches the local PDN stack and optimization layer. The solver reuses a checkpoint-derived warm start, which helps the ADMM consensus variables move toward a feasible coordinated schedule more quickly. 

{
  "backend": "admm_dispatch",
  "warm_start": true,
  "iterations": 18,
  "first_step_grid_buy": 12.7
}

Step 5. Verify and review

Verification checks line stress, power balance, and acceptance thresholds. HCII then summarizes the risk and explains why the recommended first-step action is acceptable. 

{
  "accepted": true,
  "violations": [],
  "review_note": "Use PDN-6 V2G support to relieve ramp stress while keeping EV mobility above target."
}

Step 6. Execute the validated first step

Only the first-step control is executed. This preserves the rolling nature of the loop and lets the framework re-evaluate the system after fresh measurements arrive. 

{
  "execution": "released",
  "applied_control": {"grid_buy": 12.7, "battery_discharge": 4.1, "v2g_discharge": 1.3},
  "observed_feedback": {"served_ratio": 0.996, "price_realization": 1.08}
}

Step 7. Write back accepted experience

The cycle feedback becomes reusable operating knowledge. Policy memory is updated, the accepted case is stored, and the workflow checkpoint is refreshed so the next cycle can start from a stronger prior. 

{
  "write_back": {
    "accepted_case": true,
    "policy_update": {"acceptance_threshold": 0.92},
    "checkpoint_saved": "dispatch_profile_8pdn_high_ramp"
  }
}
Prompt Logic

How prompts guide the LLM inside the closed loop

The prompt layer is written to keep the LLM useful inside an operational system. Instead of open-ended conversation, each role receives grounded inputs, explicit reasoning goals, hard safety constraints, and a machine-readable output contract.

Role boundary

Each agent prompt defines what the role is allowed to interpret and what it must leave to solvers or verification.

Input grounding

Prompts operate on explicit system state, bridge variables, memory retrieval, and feedback summaries.

Safety priority

Prompts state that feasibility, policy compliance, and risk management outrank narrative fluency.

Structured output

Every response is constrained into exact keys so downstream code can consume it deterministically.

Shared prompt shell

Across ITGC, Orchestrator, DSE, Optimization, Verification, HCII, and Execution, the prompts follow one consistent pattern. That consistency is what lets the LLM cooperate with the rest of the code rather than floating outside the runtime loop. 

Mission:
  Define the role-specific objective for this cycle.

Current inputs:
  Summarize the state, memory retrieval, bridge variables,
  solver signals, and policy context available to this role.

Reasoning requirements:
  State the exact operational questions the role must answer.

Hard constraints:
  Prioritize safety, consistency, and valid structured output.

Return JSON with exactly these keys:
  A stable schema that downstream code can parse directly.

Memory Evolution

How persistent memory makes the system smarter over time

The memory layer does not just archive past runs. It stores reusable operating knowledge that directly changes future planning, dispatch, verification, and workflow orchestration. This is what gives the framework iterative improvement without changing model weights.

State & Context Memory

Historical states, exogenous signals, trigger patterns, alarms, and recent operating context.

Policy & Goal Memory

Objective weights, acceptance thresholds, reserve bias, risk posture, and strategic preferences.

Accepted Case Memory

Successful {X_k, z_k, Gamma_k, M_k} cycles that can be reused as reference patterns.

Failure & Rollback Memory

Rejected decisions, failure labels, unsafe signatures, and rollback reasons that inform future avoidance.

Workflow & Checkpoint Memory

Feasible checkpoints, warm starts, solver traces, and runtime templates for faster future coordination.

Success-driven evolution

Reuse what worked

Accepted cases improve future bridge design, solver initialization, and strategy selection.

Failure-driven evolution

Avoid what failed

Failure records tighten verification and discourage future plans that resemble known bad contexts.

Workflow-driven evolution

Start from better checkpoints

Checkpoint memory accelerates repeated coordination by preserving feasible solver and workflow states.

File-backed memory in the running system

Memory is stored in a persistent JSON file so knowledge survives across runs. The framework reads it at cycle start and writes to it after context observation, accepted execution, rejected execution, and checkpoint updates.
